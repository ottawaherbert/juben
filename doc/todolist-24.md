## 1. 需求背景与设计思路
为了适应未来多模态、多渠道的 AI 模型接入需求，我们需要对现有的系统设置进行重构，并设计一套高扩展性的“渠道-模型-任务路由”架构。同时，精简现有不合理的菜单结构。

### 1.1 核心架构设计：渠道 (Channel) -> 模型 (Model) -> 路由 (Routing)
*   **渠道 (Channel / Provider)**: 负责定义“如何连接”。
    *   **属性**：渠道名称、API 规范 (OpenAI 通用格式 / Gemini 原生格式 / ComfyUI 本地接口 / 其他自定义)、API Base URL、默认 API Key。
*   **模型 (Model)**: 负责定义“具体能力”。
    *   **属性**：模型别名、实际 Model ID (如 `gpt-4o`, `gemini-1.5-pro`, `deepseek-chat`)、所属渠道、能力标签 (文本生成、图片生成、视频生成)、独立 API Key (可选，用于覆盖渠道默认 Key)。
*   **任务路由 (Task Routing)**: 负责定义“谁来干活”。
    *   将系统内的具体业务节点分配给指定的模型。
    *   **例如**：剧本生成 -> DeepSeek-V3 (OpenAI渠道)；分镜生成 -> GPT-4o；图片生成 -> 本地 ComfyUI (FLUX工作流)；视频生成 -> Gemini-2.5-Video。

### 1.2 ComfyUI 本地接入专项设计
*   **工作流管理**：用户在 ComfyUI 中点击 "Save (API Format)" 导出 JSON 并粘贴到系统中。
*   **参数映射 (Node Mapping)**：系统解析 JSON 节点，提供可视化界面，让用户指定系统的请求参数对应哪个 Node。
    *   例如：将系统的 `prompt` 映射到 `Node 6.inputs.text`，将 `image` (图生图) 映射到 `Node 10.inputs.image`。
