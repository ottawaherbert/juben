# 第一性原理视角下的系统制作流程梳理与优化计划 (Todolist-27)

## 1. 核心第一性原理定义 (First Principles)

**影视制作的本质是什么？**
是将创作者脑海中的**“抽象概念（Idea）”**，通过一系列标准化的媒介（文字、图像、声音），最终转化为**“具象的视听体验（Video/Audio）”**的过程。

**AI 辅助影视制作的本质是什么？**
是利用 AI 作为“放大器”和“翻译官”，降低从“概念”到“视听”每个转化环节的**技术门槛**和**时间成本**，同时保持创作者的**核心控制力（Director's Vision）**。

基于此，一个“好用且专业”的系统必须满足：
1. **数据流转无损**：上游设定的资产（角色、场景）和逻辑（剧本），必须能无缝、准确地传递给下游（分镜、视频生成），不能出现上下文断层。
2. **控制粒度渐进**：从宏观（世界观、大纲）到中观（场次、剧本），再到微观（镜头、帧、音轨），创作者可以随时介入修改，也可以放权给 AI 补全。
3. **非线性迭代**：创作不是单向流水线。在画分镜时发现剧本不对，应该能无缝切回剧本修改，且不破坏已有的分镜结构。

---

## 2. 流程处理计划与梳理方案

### 评估维度
我们将沿着当前系统的核心生命周期（**起点 -> 设定 -> 结构 -> 剧本 -> 分镜 -> 工作室**）进行扫描，重点考察以下四个维度：
1. **认知负荷 (Cognitive Load)**：用户在当前步骤是否需要记忆或手动搬运上一步的信息？
2. **专业映射 (Professional Mapping)**：当前的 UI/UX 是否符合影视工业标准（如好莱坞剧本格式、标准分镜表），同时又适应 AI 的特性？
3. **AI 效能 (AI Leverage)**：AI 是在做“填空题”（低效），还是在做“基于上下文的推演”（高效）？
4. **资产一致性 (Asset Consistency)**：角色、场景的视觉和声音特征是否在全流程中保持一致？

## 3. 当前流程诊断：不合理之处 (Pain Points)

通过对现有代码和架构的梳理，发现以下几个核心不合理之处：

### 痛点一：剧本 (Script) 与 分镜 (Storyboard) 的映射关系僵化
* **现状**：目前系统倾向于“一场戏 (Scene) -> 多个镜头 (Shot)”。剧本块（ScriptBlock）和镜头（Shot）之间缺乏直接的、细粒度的绑定关系。
* **不合理**：在专业制作中，一句台词或一个动作描写（ScriptBlock）可能对应一个镜头，也可能被拆分为正反打两个镜头。目前用户在 Storyboard 页面生成分镜时，很难精确控制“这句话对应哪个画面”，导致 AI 生成的画面与剧本脱节。
* **违背原则**：数据流转断层，微观控制力不足。

### 痛点二：资产 (Bible) 在生成环节的“失忆”问题
* **现状**：在 Bible 中定义了角色（外貌、性格）和场景。但在 Studio 或 Storyboard 调用 Imagen/Veo 生成图像/视频时，仅仅是将名字拼接到 Prompt 中。
* **不合理**：大模型对名字没有视觉记忆。如果不把 Bible 中的“视觉特征（Visual Prompt）”或“参考图（Reference Image）”强制注入到每一次生成请求中，生成的角色会长得完全不一样（角色不一致是 AI 影视的致命伤）。
* **违背原则**：资产一致性断裂。

### 痛点三：结构 (Structure/Episodes) 层级过深，操作繁琐
* **现状**：用户需要先建 Episode，再建 Scene，再写 Script。
* **不合理**：对于微短剧或单集短片（Movie），强制的 Episode -> Scene 层级显得过于笨重。很多创作者习惯直接写大纲（Beat Sheet），然后直接扩写剧本。当前的 UI 强制用户在多个页面跳跃，认知负荷高。
* **违背原则**：认知负荷高，非线性迭代困难。

### 痛点四：Studio (时间轴) 的音视频对齐逻辑过于基础
* **现状**：Studio 页面有 Timeline，支持拖拽，但音频（TTS、BGM、SFX）和视频（Shot）的对齐逻辑是松散的。
* **不合理**：在实际剪辑中，视频画面往往需要根据音频（尤其是台词 TTS）的长度来决定。目前系统似乎是先定镜头长度，再配音，这会导致“声画不对位”。
* **违背原则**：不符合专业视听制作的物理规律（通常是先有台词/动效时间轴，再铺画面）。

## 4. 解决方案与优化目标 (Proposed Solutions)

为了实现“好用且专业”，建议在下一阶段进行以下重构（待确认）：

### 方案一：建立“剧本-分镜”的锚点映射 (Script-to-Shot Anchoring)
* **设计**：在 Script 页面，允许用户选中一段文本（如一行动作描写），直接点击“生成分镜”。系统在底层建立 `Shot` 关联到特定 `ScriptBlock ID` 的映射。
* **效果**：在 Storyboard 页面，左侧显示剧本，右侧显示分镜，高亮当前镜头对应的剧本段落。做到“所见即所得”。

### 方案二：构建“全局资产注入与前端可视化引擎” (Visible Global Asset Injection)
* **设计**：
  1. **全资产覆盖**：在生成图像/视频 Prompt 时，系统自动扫描当前镜头关联的 `characterIds`、`locationId` 以及 **`propIds`（道具ID）**。同时检查并补全剧本页面（Script）对道具 ID 的提取和映射。
  2. **绝对的前端可见性（核心要求）**：提取到的 Bible 资产信息（如 `visualDescription` 或参考图）**绝对不能**在底层代码中暗中拼接。必须将其作为变量，显式地注入到前端的 Prompt 模板中，并在弹出的 `PromptEditorModal` 中完整展示。
* **效果**：彻底解决 AI 生成角色/场景/道具不一致的问题，同时保证创作者对 Prompt 拥有 100% 的知情权、修改权和固化权。

### 方案三：工作流扁平化与“快速模式”开关 (Fast Mode Toggle)
* **设计**：不将工作流与项目类型（Movie/TV/Short-Drama）写死绑定。在系统/项目设置页面增加一个**“快速模式 (Fast Mode)”**开关。
  * **开启时**：简化流程，合并 Episodes 和 Structure 页面，提供一个扁平化的“大纲板 (Beat Board)”，支持卡片式拖拽，双击直接展开写剧本。
  * **关闭时**：保持标准的工业化层级（分集 -> 场次 -> 剧本）。
* **效果**：赋予创作者选择权，既能满足重度工业化需求，也能降低轻量级创作的操作门槛。

### 方案四：以“音频轨”为驱动的智能时间轴 (Audio-Driven Timeline)
* **设计**：在 Studio 中，当生成 TTS 后，自动根据音频长度锁定或建议关联镜头（Shot）的最小 `duration`。支持“吸附 (Snap)”功能，让视频片段自动对齐台词的起止点。
* **效果**：大幅减少用户手动微调时间轴的痛苦，提升成片质量。

- [x] **Timeline 交互优化**：在 `Timeline.tsx` 中增加拖拽吸附（Snap to Audio）逻辑。
