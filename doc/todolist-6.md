# O.M.N.I. 系统深度优化与未来演进方向 (To-Do List 6)

在完成了核心数据流打通、错误边界处理、全局状态指示以及基础的时间轴功能后，O.M.N.I. 系统已经具备了“工业级创作管线”的雏形。为了进一步提升系统的专业性、稳定性和扩展性，我们需要从以下几个维度进行深度优化：

## 电视剧 (TV Series) 与 微短剧 (Short Drama) 模式深度优化方案

在深入分析了现有的工作流（起点 -> 分集 -> 节拍大纲 -> 剧本 -> 分镜）后，针对“电视剧”和“微短剧”模式，发现以下不合理之处，并提出相应的专业化改进方案：

### 1. 节拍大纲模板 (Structure) 的不匹配
**问题分析：**
目前系统提供的默认模板（《救猫咪》15节拍、英雄之旅12步、经典三幕剧）都是典型的**电影长片**结构。
- **电视剧**单集通常采用“多线叙事（A/B/C故事线）”或“美剧经典四幕/五幕剧结构（Teaser + 4 Acts + Tag）”。
- **微短剧**单集只有1-2分钟，根本无法容纳《救猫咪》的15个节拍。微短剧的核心是极快的节奏、“情绪价值”、“爽点”、“反转”和“钩子（Cliffhanger）”。

**改进方案：**
- [ ] **动态推荐默认模板**：在 `TemplateModal.tsx` 中，根据 `currentProject.type` 动态加载不同的默认模板列表。
  - **电视剧专属模板**：美剧单集五幕结构（Teaser, Act 1-4, Tag）、A/B双线叙事结构。
  - **微短剧专属模板**：微短剧经典“三翻四抖”结构、打脸逆袭结构、强悬念钩子结构。
- [ ] **微短剧生成指令优化**：在微短剧的节拍生成 Prompt 中，强制要求每集结尾必须有强烈的“钩子（Hook）”，且节奏必须极快，避免冗长的铺垫。

### 2. 分集管理 (Episodes) 的效率问题
**问题分析：**
- **微短剧**通常有 80-100 集。现在的分集列表如果展示 100 集会非常拥挤，且一集一集手动添加或生成效率极低。
- **电视剧**有“季（Season）”的概念和“整季主线（Season Arc）”，目前的单集生成虽然有前后集上下文，但缺乏对整季最终目标的宏观把控。

**改进方案：**
- [ ] **微短剧批量生成与分组**：在 `Episodes.tsx` 中，为微短剧增加“批量生成（如一次生成 10 集大纲）”的功能；支持按“卷”或“篇章”（如 1-10集，11-20集）进行分组折叠管理。
- [ ] **电视剧季主线注入**：在项目全局信息中增加“季主线（Season Arc）”字段，让单集生成时不仅参考前后集，还能紧扣整季的最终目标。

### 3. 剧本生成 (Script) 的风格差异
**问题分析：**
微短剧剧本重台词、重动作、轻环境描写，且多为竖屏拍摄（9:16）。目前的 Prompt 可能会生成传统的、偏重环境描写的电影剧本。

**改进方案：**
- [ ] **微短剧剧本 Prompt 优化**：在 `Script.tsx` 的 AI 生成剧本 Prompt 中，如果检测到是微短剧，则加入指令：“请采用微短剧剧本风格：台词精炼直接、冲突密集、动作夸张、减少不必要的环境描写。注意这是竖屏拍摄，多用特写和近景。”
- [ ] **电视剧剧本 Prompt 优化**：强调人物关系的拉扯和多线交织的自然过渡。

### 4. 分镜生成 (Storyboard) 的构图逻辑
**问题分析：**
竖屏（9:16）的构图逻辑与横屏完全不同，极远景（EWS）在竖屏中表现力极差，而微短剧几乎都是竖屏。

**改进方案：**
- [ ] **竖屏分镜指令优化**：在 `Storyboard.tsx` 中，如果项目是微短剧（或检测到 `aspectRatio` 为 9:16 等竖屏比例），在生成分镜 Prompt 中强制要求：“由于是竖屏拍摄，请多使用特写（CU）、中近景（MCU），尽量减少大远景（EWS），确保主体占据画面核心，适应手机屏幕的观看习惯。”

## 第一阶段：数据持久化与云端同步 (架构升级)
当前系统依赖 `localStorage` 进行数据存储，这在处理大量图片、视频和音频资产时极易触发浏览器存储限制（通常为 5MB），且存在数据丢失风险。

- [ ] **迁移至 IndexedDB**：使用 `localforage` 或 `idb` 替换现有的 `localStorage` 持久化方案，突破 5MB 限制，支持大规模媒体资产的本地缓存。
- [ ] **项目导入与导出**：实现将整个项目（包含 JSON 数据和 Base64 媒体文件）打包导出为 `.omni` 或 `.zip` 文件的功能，并支持离线导入。

## 第二阶段：摄影棚 (Studio) 进阶合成与渲染
目前的合成功能仅为前端的简单预览，无法真正输出成片。

- [x] **FFmpeg.wasm 纯前端合成**：利用已安装的 `@ffmpeg/ffmpeg`，在浏览器端将视频片段、配音、BGM 和音效按照 Timeline 的时间戳进行精准混合，并导出为 MP4 文件。
- [x] **多镜头连贯导出**：支持将当前集数 (Episode) 下的所有分镜 (Shots) 按照顺序拼接，生成完整的视频流。
- [x] **转场特效支持**：在时间轴的镜头交界处，允许添加简单的转场效果（如淡入淡出、黑场过渡）。

## 第三阶段：剧本 (Script) 编辑器的专业化
当前的剧本编辑器较为基础，难以满足复杂剧本的精细化控制需求。

- [x] **块级 (Block-based) 剧本编辑器**：参考 Notion 或专业编剧软件，将剧本拆分为独立的“场景块”、“动作块”和“台词块”，支持拖拽排序。
- [x] **AI 局部重写与扩写**：允许用户选中某一句台词或某一个动作，单独调用 AI 进行“语气调整”、“扩写细节”或“缩写”。
- [x] **情绪与镜头语言智能标注**：在剧本生成时，AI 自动为每句台词标注情绪（如：愤怒、悲伤），为每个动作标注建议的镜头景别（如：特写、全景），并直接联动到分镜生成和 TTS 配音。

## 第四阶段：资产库 (Assets) 与提示词 (Prompt) 工程
- [x] **高级 Prompt 模板管理**：在设置页面提供 Prompt 模板编辑器，允许高级用户自定义生图、生成视频和生成剧本的系统级 Prompt。
- [x] **资产标签与检索系统**：当资产库变得庞大时，支持为角色、场景添加标签（如：赛博朋克、中世纪、反派），并提供快速检索功能。

## 第五阶段：响应式与性能优化
- [ ] **移动端/平板端适配**：优化侧边栏、时间轴和画布在小屏幕设备上的显示效果，确保在 iPad 等设备上也能流畅进行创作。
- [ ] **媒体资源懒加载**：在全部分镜列表和资产库中，对图片和视频实施懒加载（Lazy Loading），降低初次渲染的内存占用和卡顿感。
- [ ] **Web Worker 卸载计算**：将复杂的 AI 数据解析、实体对比等逻辑移至 Web Worker 中执行，避免阻塞主线程 UI。
